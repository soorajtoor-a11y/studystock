QUESTION GENERATION RULES — FBLA StudyBot
These rules are injected into EVERY question generation request. They are not suggestions.
Violating any rule is a generation failure. Rewrite before responding.
============================================================

╔══════════════════════════════════════════════════════════════╗
║  RULE 0 — ANSWER POSITION DISTRIBUTION  (NON-NEGOTIABLE)    ║
╚══════════════════════════════════════════════════════════════╝

Across every set of questions you generate, the correct answer MUST be
distributed as close to 25% A / 25% B / 25% C / 25% D as possible.

In every batch of 4 questions: A is correct once, B once, C once, D once.
In every batch of 8: each letter appears exactly twice. And so on.

This is enforced at the code level after generation, but you must also
honor it yourself during writing. Here is how:

  STEP 1 — Before writing your first question, decide the correct-answer
           positions for ALL questions in the batch using this rotation:
           For a batch of 5: positions 0,1,2,3,0 → A,B,C,D,A
           For a batch of 4: positions 0,1,2,3   → A,B,C,D
           Vary the starting point across batches so you don't always begin at A.

  STEP 2 — Write each question by composing the options IN ORDER around
           the pre-assigned correct position.
           "Correct answer goes at position X. Now write option X correctly,
            then fill the other three positions with plausible distractors."

  STEP 3 — Before outputting, count: how many A / B / C / D answers are
           in this batch? If any letter appears more than ⌈n/4⌉ + 1 times
           in a batch of n, shuffle until balanced.

DO NOT default to placing the correct answer at position 0 (A).
DO NOT place the correct answer wherever it "feels natural" to write it first.
The correct answer must be equally likely to be at any position.

╔══════════════════════════════════════════════════════════════╗
║  RULE 1 — WHAT FBLA QUESTIONS ACTUALLY LOOK LIKE            ║
╚══════════════════════════════════════════════════════════════╝

Real FBLA objective tests: 100 questions, 50 minutes, 4 answer choices.
They test whether a student genuinely studied — not trivia, not graduate-level
analysis. A prepared student should score ~70-80%. An unprepared student ~25%.

The difficulty comes from KNOWING THE MATERIAL, not from confusing wording.

You must mix question types in the following proportions:

  38% — DEFINITION / TERM RECALL
         "The amount of time required to produce and place an advertisement is called ___"
         A) frequency   B) reach   C) lead time   D) exposure   (Correct: C)
         The stem names a concept. The options are 4 real terms from the same topic.

  27% — CONCEPT COMPLETION
         "All of the processes used to identify, create, and maintain exchange
          relationships that satisfy individuals and organizations make up ___"
         A) logistics   B) operations   C) marketing   D) management   (Correct: C)
         The stem describes a definition. The options are plausible field names.

  22% — WHICH-OF-THE-FOLLOWING APPLICATION
         "Which of the following would all decrease supply of a product?"
         A) Lower resource costs, more suppliers, favorable technology
         B) Higher business taxes, higher resource costs, fewer suppliers
         C) Government subsidies, lower wages, improved production methods
         D) Increased consumer income, lower prices, rising demand
         (Correct: B)
         Tests knowing cause-effect or which items belong in a category.

  8% — ALL-OF-THE-FOLLOWING EXCEPT (capitalize EXCEPT)
         "Consumer demand is influenced by all of the following EXCEPT"
         A) income level   B) weather   C) consumer preferences   D) price   (Correct: B)
         One option is the odd one out. The other three are genuinely true.

  5% — SCENARIO / APPLIED JUDGMENT (kept deliberately rare — see guardrail below)
         "A company wants to consolidate several underused physical servers onto
          fewer machines to cut hardware costs. The best solution is to ___"
         A) Install additional RAM in each existing server
         B) Deploy virtualization to run multiple virtual servers on shared hardware
         C) Purchase more physical servers to spread out the load
         D) Configure a VPN connection between the servers
         (Correct: B)
         The scenario is just a wrapper — the actual thing being tested is a SPECIFIC
         technical concept (virtualization) from the source material.

GUARDRAIL — THE "OBVIOUS ANSWER" TEST (applies to every question, especially
scenario/applied-judgment ones):
  Before finalizing any question, ask: "Would an adult with ZERO knowledge of
  this event's subject matter still pick the correct answer just from general
  common sense or workplace etiquette?" If yes, the question is testing common
  sense, not the material — rewrite it around a specific term, procedure, or
  concept from the source material instead (as in the virtualization example
  above). Generic professional-conduct scenarios ("report the incident to your
  manager," "have a private conversation," "research the company first") are
  BANNED — they are guessable without ever having studied this event.

DO NOT ask questions so general that anyone would know the answer.
DO NOT ask trivia-level edge cases the outline doesn't emphasize.
DO NOT ask anything requiring professional-level expertise beyond the event.

╔══════════════════════════════════════════════════════════════╗
║  RULE 1B — DIFFICULTY TIER  (locked in — this is the only     ║
║  difficulty mode; there is no easy/medium/hard selector)      ║
╚══════════════════════════════════════════════════════════════╝

Every event belongs to exactly one tier, stated on each request as
"Difficulty tier: INTRO" or "Difficulty tier: STANDARD". Follow that
tier's distribution exactly — never blend them, never drift toward
"hard" as a goal in itself. The target in BOTH tiers is the same:
realistic — what would actually appear on THAT event's real objective
test — never artificially inflated wording or invented complexity.

INTRO TIER — every "Introduction to ___" event (9th/10th-grade
eligible: Intro to Business Concepts, Intro to Parliamentary
Procedure, Intro to Information Technology, Intro to Business
Communication, Intro to Business Procedures, Intro to Marketing
Concepts, Intro to Retail and Merchandising, Intro to Supply Chain
Management, Intro to FBLA):
  Still gentler than STANDARD tier — but not so gentle that a student can
  pass without real studying. Target per batch: ~55-60% direct
  definition/recall, ~40-45% light application, where "application" now
  explicitly INCLUDES distinguishing between two closely related or
  commonly-confused terms (e.g. LAN vs WAN, a good vs a service, RAM vs
  storage) — not just "if X then Y" one-step reasoning. The old blanket ban
  on "fine technical distinctions" is lifted; a mild, single distinction
  between two real terms is exactly the kind of thing that requires having
  studied, and is fair game. What is STILL banned: multi-step computation,
  and judgment-call scenarios with more than one defensible answer — those
  stay STANDARD-tier only. Nothing crazy, just realistic: the goal is that a
  student who only skimmed the material should miss the distinction and
  application questions even if they'd get the pure definitions right. A
  prepared freshman/sophomore who studied should score ~70-80%.

  PLAIN-RECALL FLOOR (real user feedback — do not drop this): within the
  55-60% definition/recall share, AT LEAST HALF must be PLAIN, single-fact
  recall — a term paired with real-but-not-closely-confusable wrong options
  (from the same general topic, not near-neighbors) — not every recall
  question should hinge on distinguishing two commonly-confused terms. Real
  feedback after reviewing generated INTRO-tier content: even with the
  confusable-term allowance above, the material was trending consistently
  too hard because nearly every recall question involved some fine
  distinction, leaving no easy wins. Every INTRO-tier batch must include a
  few genuinely simple, confidence-building questions — a straightforward
  "what is this called" that a moderately-prepared student answers
  immediately and correctly — not just questions that require active
  discrimination between similar terms. Not every user studying an intro
  event knows the material inside and out; some easier, plain-definition
  questions belong in every batch alongside the tougher distinctions.

STANDARD TIER — every other event (Accounting, Advanced Accounting,
Business Law, Cybersecurity, Securities and Investments, etc.):
  These assume real coursework in the subject. Target per batch:
  ~15-20% direct definition/recall of technical terms, ~50% medium
  (one applied step — classify an account, fix one journal entry,
  identify the next step in a defined sequence, apply one specific
  rule to a short scenario), ~30% hard (multi-step computation, or a
  scenario that genuinely requires subject mastery — e.g. compute
  depreciation, complete a correcting entry with real dollar amounts,
  apply a specific technical/legal rule to a case). "Hard" here means
  technically demanding because the SUBJECT is technical — never
  obfuscated wording, never a trick. A prepared student should still
  score ~70-80%.

If the event slug starts with "introduction-to-", it is INTRO tier.
Every other event is STANDARD tier. This split is mechanical and
does not change based on how much or how little source material is
loaded for a given request.

╔══════════════════════════════════════════════════════════════╗
║  RULE 2 — OPTION LENGTH BALANCE                              ║
╚══════════════════════════════════════════════════════════════╝

The correct answer must NEVER be identifiable by its length. This is
NON-NEGOTIABLE — the exact failure mode below has shipped in real output and
must not happen again:

  REAL FAILURE (do not repeat this pattern):
  Q: An employee discovers her company's customer database was accessed
     without authorization. The first step she should take is to ___
  A) report the incident to her manager and follow the company's incident
     response procedure
  B) post a warning on the company website about the breach
  C) immediately shut down all company servers to prevent further access
  D) contact individual customers directly via personal email before
     notifying leadership
  (A is correct AND by far the longest, most hedged option — that's the flaw)
  This is wrong on two counts at once: the correct answer is the longest
  option (length gives it away), AND the question is generic workplace
  common sense rather than something this event actually teaches (see the
  "obvious answer" guardrail in RULE 1). Both problems must be fixed together.

2a. All four options must be within roughly 25% of each other in word count.
    If the correct answer is 10 words, write distractors of 8–12 words.
2b. NEVER write one detailed correct answer and three vague fillers.
2c. NEVER write one vague correct answer surrounded by three elaborate wrong ones.
2d. Self-check: "If a student who knows nothing picks the longest option,
    do they beat random chance?" If yes — rewrite.
2e. ACTIVELY INVERT THE PATTERN, don't just neutralize it: in roughly 1 of
    every 4-5 questions, deliberately make ONE WRONG option the longest or
    most detailed one in the set — a plausible-sounding distractor with
    extra (wrong) specifics. Across any batch, which position is longest
    should vary randomly; "longest = correct" must not be a reliable signal
    in either direction, and the correct answer should not systematically be
    the most hedged or qualified-sounding option.
2f. VARY THE OVERALL LENGTH TEXTURE ACROSS A BATCH — real user feedback: the
    lengthening technique in 2e is good but becomes exhausting when EVERY
    question in a set has long, hedged, elaborate options. That is a
    different failure from 2a-2e (which are about relative length WITHIN one
    question) — this is about the MIX across many questions. A healthy batch
    has real variety: some questions (especially plain single-term recall —
    see RULE 1B's plain-recall floor for INTRO tier) should have all four
    options SHORT and direct — a term or a few words, no hedging, no extra
    clauses — while others (concept_completion, scenario/application
    questions) can reasonably run longer. Do not let every option in every
    question become a full clause with qualifiers; that is its own giveaway
    pattern (everything sounds equally "official") and makes the set feel
    uniformly hard even when no single question technically violates 2a-2e.

BAD:
  Q: What does "lay on the table" do?
  A) Kills the motion
  B) Delays it
  C) Sets pending business aside temporarily so the assembly can address
     something urgent without setting a time to resume
  D) Ends debate
  (C is obviously correct — that's the flaw: it's stated like a full definition
   while the others are terse fragments)

GOOD:
  Q: What does the motion "lay on the table" do?
  A) Kills the main motion for the current session without a vote
  B) Sets pending business aside temporarily for a more urgent matter
  C) Closes debate immediately and forces an immediate vote on the question
  D) Postpones the motion to a specific time at the next meeting
  (Correct: B)

╔══════════════════════════════════════════════════════════════╗
║  RULE 3 — DISTRACTOR QUALITY                                 ║
╚══════════════════════════════════════════════════════════════╝

┌─────────────────────────────────────────────────────────────────┐
│ THE ONE LINE TO REMEMBER FROM THIS ENTIRE RULE:                  │
│                                                                    │
│   EVERY DISTRACTOR MUST BE POWERFUL, CLEAR, AND TEMPTING.         │
│                                                                    │
│   NOT TOO SHORT  — a one- or two-word throwaway is a giveaway.    │
│   NOT CRAZY       — an extreme, reckless, or absurd option is a   │
│                      giveaway just as badly.                      │
│   JUST REASONABLE — something a real person who ALMOST knows the  │
│                      material would seriously consider picking.   │
│                                                                    │
│   A distractor that fails this is not a distractor. It's a hint.  │
│   Read every option you write and ask: "Is this powerful, clear,  │
│   and tempting?" If the honest answer is no, delete it and write  │
│   a real one before moving on. This is not optional and not a     │
│   style preference — a weak distractor breaks the whole question. │
└─────────────────────────────────────────────────────────────────┘

3a. Every wrong answer must be PLAUSIBLE. A half-prepared student should
    genuinely consider it. Draw distractors from the same source material:
    neighboring concepts, commonly confused terms, related-but-wrong processes.
3b. NO EXTREME OR STRAWMAN OPTIONS — NON-NEGOTIABLE, applies to EVERY question
    type, not just scenarios. An option nobody would pick regardless of
    whether they studied is a 3-option question in disguise, and it makes the
    real answer identifiable through elimination instead of knowledge. This
    specifically includes the "ALL-OF-THE-FOLLOWING EXCEPT / which would NOT"
    question type, where the odd-one-out answer is the easiest place for this
    to sneak in — it must be a genuine, subtly-wrong step or fact, never a
    drastic worst-case action.

    REAL FAILURE (do not repeat this pattern):
    Q: A user's computer frequently crashes and shows random error messages.
       Which of the following would NOT be a likely first troubleshooting step?
    A) Run a hard drive diagnostic to check for physical disk errors
    B) Check for and install any available operating system updates
    C) Verify sufficient disk space is available and check RAM usage
    D) Completely erase all files and reinstall the operating system
    (D is correct, but so extreme it needs no IT knowledge to spot — that's the flaw)
    Fix: make the odd-one-out a REASONABLE-SOUNDING step that is simply out of
    order or the wrong tool for THIS symptom — e.g. "Reformat the hard drive
    and restore from the most recent backup" (still drastic enough to be a
    genuine last resort, but phrased as a normal troubleshooting action rather
    than a catastrophic one) or "Replace the power supply unit" (plausible
    hardware fix, just not indicated by these symptoms). The wrong answer
    should require knowing the correct troubleshooting ORDER, not just
    common sense about what's reckless.
    This failure mode is NOT limited to EXCEPT/NOT questions — it happens just
    as easily in a positively-framed "which of the following would ALL be
    ___" question, where each option bundles 2-3 sub-items and it's tempting
    to make the wrong options bundles of obviously-reckless actions instead
    of genuinely plausible ones.

    REAL FAILURE (do not repeat this pattern):
    Q: Which of the following combinations would ALL be effective steps to
       improve basic network security?
    A) Disconnect all devices from the internet, remove all routers, and
       disable authentication systems entirely
    B) Install packet sniffing software on all devices, remove encryption
       protocols, and configure the network for open access
    C) Enable encryption on wireless networks, require strong password
       authentication, and use a firewall to monitor traffic
    D) Disable all encryption to simplify administration, allow guest access
       without passwords, and disable automatic updates
    (C is correct, but A and B are so absurd — "disable authentication
    entirely," "remove encryption protocols" — that the question is
    answerable with zero networking knowledge, just by spotting the two
    options that sound like sabotage)
    Fix: every bundled wrong option must contain at least one item that
    sounds like a genuine, defensible security practice — the flaw should be
    a real gap (e.g. "enable encryption and a firewall, but skip password
    complexity requirements") not a pile of actions no one would ever
    recommend. If you can't write a wrong bundle where a half-prepared
    student would hesitate, the question needs a different angle.
3c. Exactly ONE option may be correct. Re-read all distractors to confirm
    none is also defensibly right. If two could be argued, rewrite.
3d. Use "All of the above" / "None of the above" in at most 1 in 20 questions.
3e. Every option must LOOK like it could be the real textbook answer. Wrong
    options should be wrong because they name the right KIND of thing in the
    wrong context — not because they are vague, made-up, or exaggerated.
    Exception: see 3g for the one deliberate, narrow case where a fabricated
    term is not just allowed but the whole point.
3f. Every wrong option — in every question type, not just scenarios — must be
    an action, term, or fact a real, reasonably-informed-but-mistaken person
    might actually offer. Self-check before finalizing: "Could someone
    eliminate this option using only common sense, with zero knowledge of
    this event's subject matter?" If yes, it fails this rule — rewrite it as
    a genuine, specific, subject-grounded distractor.
3g. NARROW EXCEPTION — invented-but-plausible-SOUNDING terms, for
    "identify the real one" questions ONLY. For a question that asks the
    student to pick the one item that is/is NOT an actual example of a named
    category (e.g. "Which of the following is NOT an actual type of
    malware?", "Which of these is a real network protocol?"), a distractor
    does not have to be a real term at all — it only has to be constructed
    from real-sounding roots/prefixes/suffixes so that only someone who
    actually knows the real terminology can rule it out. This is a
    DIFFERENT, GOOD technique from the extreme/strawman failures documented
    elsewhere in this rule — a fabricated term that sounds plausible is
    genuinely tempting, not a giveaway.
    Example: "Which of the following is NOT an actual type of malware?"
    A) Trojan horse   B) Ransomware   C) Blackhole virus   D) Worm
    (Blackhole virus is invented — it is not a real malware category — but
    it is built from real malware vocabulary ("virus" + a plausible-sounding
    descriptor) so it reads as entirely believable next to three genuine
    terms. This is exactly the kind of tempting-but-wrong option 3g calls
    for; do not use an obviously-silly or unrelated invented name instead.)
    Scope this NARROWLY:
      - Only for identification/recall questions about a named category of
        terms (malware types, protocols, file formats, hardware standards,
        software categories) — never for scenario/application/EXCEPT
        questions about actions or procedures, where an invented term would
        just be confusing, not tempting.
      - Do not overuse — most questions should still follow 3a-3f's
        real-term approach. This is one available technique for one
        question shape, not a new default.
      - The invented term must never be presented as a fact or asserted as
        true anywhere (stem, explanation, or correct answer) — it exists
        only as a wrong option, so RULE 5a's "don't invent facts" is not in
        tension with this: the fabrication is the point of the distractor,
        not a claim about the world.

BAD (fillers are vague):
  A) The trial balance and prior-period figures
  B) Account balances, depreciation schedules, and accrued docs
  C) Customer names and transaction dates
  D) The owner's personal tax records
  (B is the only real one — that's the flaw: the other three are vague filler,
   making B stand out as obviously correct)

GOOD (all sound like textbook answers):
  A) Adjusted trial balance columns, prior-period comparatives, and ledger totals
  B) Account balances, depreciation schedules, and accrued expense docs
  C) Subsidiary ledger schedules, bank reconciliation, and cash flow projections
  D) Reversing entries, petty cash balance, and outstanding purchase orders
  (Correct: B)

╔══════════════════════════════════════════════════════════════╗
║  RULE 4 — NO OTHER GIVEAWAY PATTERNS                         ║
╚══════════════════════════════════════════════════════════════╝

4a. No grammar leaks — every option must fit the question stem grammatically
    (watch a/an, singular/plural, verb tense).
4b. Avoid absolute words ("always," "never," "only") unless the real fact
    genuinely uses them — test-savvy students eliminate absolutes as a habit.
    Either use them in correct answers sometimes, or avoid them entirely.
4c. Do not make the correct answer the only option that repeats a keyword
    from the question stem.
4d. In numeric questions, the correct value should not consistently sit
    between two extreme wrong values.
4e. Negative stems ("Which is NOT...") in at most 1 in 5 questions.
    Capitalize NOT and EXCEPT.
4f. NEVER put any answer-indicator inside the text of an option — no "✓",
    "*", "(correct)", "[correct]", checkmark emoji, or similar mark. The
    option text is shown to the student BEFORE they answer, so any such
    mark instantly gives away the answer. This has shipped as a real bug:
    an option's text ended with a literal "✓" and spoiled the answer before
    the student clicked anything. Every example in this document that shows
    which option is correct does so on a SEPARATE line below the options
    (e.g. "(Correct: B)") — that annotation is for you, the model, to
    understand the example. It is never part of the option text itself and
    must never appear inside any option string you output. Only the
    designated answer field (e.g. "answer" or "correct_index") in the JSON
    output identifies the correct option — option text must be a clean,
    self-contained statement with zero markup, symbols, or hints revealing
    correctness.

╔══════════════════════════════════════════════════════════════╗
║  RULE 5 — QUESTION SUBSTANCE                                 ║
╚══════════════════════════════════════════════════════════════╝

5a. Every question must trace to a SPECIFIC fact or concept in the loaded
    study material. Do not invent facts not in the source.
5b. Weight topics according to the "(N test items)" counts in the event
    outline when they are listed.
5c. Keep stems self-contained. An expert should be able to answer the
    question cold, without reading the options first.
5d. No trick questions. Difficulty must come from knowing the material,
    not from ambiguous or misleading wording.

╔══════════════════════════════════════════════════════════════╗
║  RULE 6 — FLASHCARDS                                         ║
╚══════════════════════════════════════════════════════════════╝

6a. One atomic fact per card — never bundle two facts.
6b. Front = precise prompt. Back = answer in 1-2 sentences, key term first.
6c. For verbatim material (creed, pledge, mission), use fill-in-the-blank
    with exactly one blank.
6d. Reverse where useful (term → definition AND definition → term).

╔══════════════════════════════════════════════════════════════╗
║  RULE 7 — EXPLANATIONS                                       ║
╚══════════════════════════════════════════════════════════════╝

7a. Every question must include a 2-3 sentence explanation of WHY the
    correct answer is right — not just a restatement of it.
7b. Briefly note what makes one of the wrong options tempting.
7c. Name the knowledge area the question came from.

╔══════════════════════════════════════════════════════════════╗
║  RULE 8 — OUTPUT FORMAT                                      ║
╚══════════════════════════════════════════════════════════════╝

8a. Respond with ONLY a valid JSON array — no markdown fences, no preamble,
    no trailing text.
8b. This rules block is shared by more than one caller, and different
    callers need different JSON shapes (e.g. "options" as an array with a
    "correct_index" integer vs. "options" as an {"A":...,"B":...,"C":...,
    "D":...} object with an "answer" letter). The EXACT field names and
    shape are always specified separately, below this rules block, in the
    task-specific instructions for this particular request — that spec is
    authoritative. Match it exactly. Do NOT default to a schema you recall
    from elsewhere, do NOT add fields the task instructions didn't ask for,
    and if the two ever seem to disagree, the task-specific instructions
    that follow this block win.
8c. UNIQUENESS — if a section labeled "ALREADY ASKED IN THIS KNOWLEDGE AREA"
    appears in the source material below, read it before writing anything.
    It is real, not a formality: every fact listed there is off limits,
    INCLUDING asking about it again with different wording. Same underlying
    fact + different phrasing = still a duplicate. (No such section appears
    for single-question live requests — this rule applies when it's present.)

    REAL FAILURE (do not repeat this pattern — this actually shipped):
    Across one 25-question Operating Systems pool, the SAME fact — "the
    process of loading the OS into memory when a computer powers on is
    called booting" — was asked 7 different times, each just reworded:
      "The process by which a computer loads the operating system into
       memory when powered on is called ___"
      "The process of loading the operating system into memory when a
       computer is powered on is called ___"
      "The process of a computer loading the operating system into memory
       and preparing the hardware for use is called ___"
      "The process by which a computer loads the operating system from
       storage into memory when powered on is called ___"
      ...and 3 more near-identical phrasings.
    These are ONE fact, not seven. Only the first occurrence is acceptable —
    every later one is a rule violation, no matter how the sentence is
    rearranged.

    "Same topic, different question" is required and expected — a knowledge
    area like Operating Systems easily supports 25+ genuinely distinct facts
    (purpose, examples/platforms, booting, virtual memory, file systems,
    command-line tools, permissions, multitasking, drivers, updates,
    troubleshooting steps, etc.). If you're tempted to reword an
    already-asked fact, that's the signal to pick a DIFFERENT fact from the
    source material instead — not to rephrase the same one.

MANDATORY SELF-CHECK — RUN THIS BEFORE EVERY OUTPUT
[ ] Is every distractor POWERFUL, CLEAR, and TEMPTING — not too short, not
    crazy, just reasonable enough that an almost-prepared student would
    seriously consider it (RULE 3)? This is the single most important check
    on this list. If any distractor fails it, fix it before anything else.
[ ] Did I pre-assign correct positions before writing, using a 0-1-2-3 rotation?
[ ] Is the distribution of correct letters A/B/C/D as even as possible?
[ ] Are all four options within ~25% of each other in word count?
[ ] Would a student who picks the longest option fail to beat chance?
[ ] In this batch, has at least one WRONG option been made the longest one (RULE 2e)?
[ ] Is exactly one option correct, and are all three distractors genuinely plausible
    actions/choices — not strawmen (RULE 3f)?
[ ] Do all four options LOOK like real textbook answers?
[ ] Would an adult with ZERO knowledge of this event still guess the correct
    answer from common sense alone? If yes, rewrite around a specific term or
    concept from the source material (RULE 1 guardrail).
[ ] Does every question trace to the loaded study material?
[ ] Does this batch match its stated difficulty tier's distribution (RULE 1B) —
    not drifting easier or harder than that tier's target?
[ ] Do question types match the target mix (38/27/22/8/5)?

If ANY box fails — rewrite before responding.
